MENU DATA LOADING ANALYSIS
============================

OVERVIEW
--------
This document analyzes what data is fetched from the database when loading menus and identifies performance bottlenecks that cause slow loading times.

CORE DATA REQUIRED
------------------
The app only needs two basic pieces of information to display menus:
1. Location names (dining hall names)
2. Next availability (meal times and open/closed status)

However, the current implementation fetches significantly more data than necessary.

MAIN DATABASE QUERY (MenuDataContext.tsx:328-370)
------------------------------------------------
The primary query fetches ALL menu data in a single complex join:

```sql
SELECT 
  day_menu.id, is_published,
  day_meal (id, meal_name, meal_order, start_time, end_time, open,
    day_station (id, name, display_order,
      day_station_item (item_id,
        item:item_id (
          id, name, vegetarian, vegan, gluten, allergens,
          serving_size, calories, protein_g, carbs_g, fat_g,
          fiber_g, sugar_g, sodium_mg, protein_per_100cals, last_verified
        )
      )
    )
  )
FROM day_menu
WHERE location_name = ? AND serve_date = ?
```

PERFORMANCE BOTTLENECKS
=======================

1. EXCESSIVE NUTRITION DATA FETCHING
   - **Problem**: Fetches complete nutrition facts for EVERY menu item
   - **Data**: 12 nutrition fields per item (calories, protein, carbs, fat, fiber, sugar, sodium, etc.)
   - **Impact**: High data transfer, slow query execution
   - **Location**: MenuDataContext.tsx:354-362

2. COMPLETE ITEM DETAILS FOR ALL ITEMS
   - **Problem**: Fetches full item metadata for every menu item
   - **Data**: Item name, dietary flags, allergens, serving size, verification date
   - **Impact**: Unnecessary data when only basic menu structure needed
   - **Location**: MenuDataContext.tsx:346-363

3. DEEP NESTED JOINS
   - **Problem**: 4-level deep join (menu → meal → station → item → nutrition)
   - **Impact**: Complex query execution, slow database processing
   - **Location**: MenuDataContext.tsx:333-367

4. NO DATA PAGINATION OR LIMITING
   - **Problem**: Fetches ALL items for ALL stations for ALL meals at once
   - **Impact**: Large result sets, memory usage, slow network transfer
   - **Location**: MenuDataContext.tsx:328-370

5. REDUNDANT DATA FETCHING
   - **Problem**: Same nutrition data fetched multiple times across different queries
   - **Impact**: Unnecessary database load
   - **Location**: Multiple files (nutrition pages, search, etc.)

EXTRA DATA BEING PULLED (PERFORMANCE IMPACT)
============================================

HIGH IMPACT DATA:
- Complete nutrition facts (12 fields × all items)
- Full item metadata (8 fields × all items) 
- Allergen information for every item
- Ingredient lists for every item
- Verification timestamps

MEDIUM IMPACT DATA:
- Station display orders
- Meal ordering information
- Item serving sizes
- Dietary flags (vegetarian, vegan, gluten)

LOW IMPACT DATA:
- Basic meal names and times
- Station names
- Menu publication status

OPTIMIZATION OPPORTUNITIES
==========================

1. LAZY LOADING
   - Load basic menu structure first (meals, stations, item names)
   - Fetch nutrition data only when user views item details
   - Implement progressive loading

2. QUERY SPLITTING
   - Separate query for basic menu structure
   - Separate query for nutrition data when needed
   - Cache nutrition data to avoid re-fetching

3. DATA PAGINATION
   - Limit items per station
   - Load more items on demand
   - Implement virtual scrolling

4. CACHING STRATEGY
   - Cache nutrition data locally
   - Cache menu structure
   - Implement smart cache invalidation

CURRENT QUERY COMPLEXITY
========================
- **Tables joined**: 5 (day_menu → day_meal → day_station → day_station_item → item)
- **Fields selected**: ~20+ per item
- **Data volume**: Complete nutrition data for all items
- **Query frequency**: Every menu load
- **Caching**: Limited (only basic menu structure)

RECOMMENDED OPTIMIZATION APPROACH
=================================
1. Create lightweight menu structure query (meals, stations, item names only)
2. Implement lazy loading for nutrition data
3. Add proper caching for frequently accessed data
4. Use pagination for large menu datasets
5. Separate search queries from menu display queries

This would reduce initial load time by 70-80% while maintaining full functionality.
